Инжектированный шеллкод в `powershell.exe`

## TL DR

В ходе расследования выявлен скрытый процесс `powershell.exe`, внутри которого исполняется инжектированный шеллкод в нескольких регионах памяти с правами `PAGE_EXECUTE_READWRITE` (RWX). Анализ этих областей показал наличие низкоуровневого x86‑кода, декодера полезной нагрузки и работы с аппаратными портами ввода‑вывода, что не соответствует нормальному поведению PowerShell и указывает на вредоносную активность.

---

## Контекст инцидента

### Исходная ситуация

- На рабочей станции замечена аномальная активность `powershell.exe`:
  - длительно работающий процесс без очевидных административных задач;
  - периодические всплески CPU/сетевой активности;
  - отсутствие понятной команды/скрипта, инициировавшего запуск.
- Параллельно обнаружены признаки использования PowerShell в подозрительных сценариях:
  - обфусцированные команды;
  - EncodedCommand с длинными Base64‑строками;
  - взаимодействие с внешними ресурсами.

### Цели расследования

- Проверить, исполняется ли вредоносный код **внутри** процесса `powershell.exe`.  
- Выявить наличие **in‑memory payload** (шеллкода), исполняемого без записи на диск.  
- Зафиксировать артефакты и сформировать боевой кейс для DFIR/Blue Team.

---

## Сбор и анализ данных

### Снятие дампа памяти

- Инструменты: `procdump`, WinDbg/x64dbg, EDR‑агент.
- Получен полный дамп процесса `powershell.exe` (PID условно `3672`).

### Анализ регионов памяти

- Через WinDbg/Volatility/x64dbg получен список VAD/Memory Regions.
- Фильтрация по правам:
  - особый интерес представляют регионы с `PAGE_EXECUTE_READWRITE` (RWX), где возможна in‑memory нагрузка.
- Выявлены подозрительные RWX‑области:
  - `0x13e0000`, `0x21a0000`, `0x9320000`;
  - `0x7ff50000`;
  - `0x7ff60000`.

---

## Разбор RWX‑регионов

### Регионы 0x13e0000 / 0x21a0000 / 0x9320000: декодер и работа с буферами

Наблюдения:

- Подготовка регистров и стека (`push es`, `cdq`, `push esp`), нетипичная для прикладного кода.  
- Многократные операции над байтами в памяти (`adc/add byte ptr [reg], al`), двигаются указатели (`add esi, ebp`).  
- Присутствуют `retf`, работа с сегментами, редкие инструкции ввода‑вывода (`out dx, al`).

Интерпретация:

- Код ходит по буферу, модифицирует байты и использует результат далее.
- Поведение соответствует **декодеру/обфускатору**, готовящему полезную нагрузку к исполнению в памяти.

### Регион 0x7ff50000: патчинг памяти и «магические» адреса

Фрагмент:

```text
3672 powershell.exe 0x7ff50000 0x7ff5ffff VadS PAGE_EXECUTE_READWRITE ...

0x7ff50000: add byte ptr [eax], al
0x7ff50002: add byte ptr [eax], al
0x7ff50004: xchg edi, eax
0x7ff50005: sbb dword ptr [eax], eax
0x7ff50007: add byte ptr [eax], al
0x7ff50009: add byte ptr [eax], al
0x7ff5000b: add byte ptr [esi], cl
0x7ff5000d: add byte ptr [eax], al
0x7ff5000f: add byte ptr [eax], ch
0x7ff50012: add byte ptr [eax], al
0x7ff50014: add cl, ch
0x7ff50016: xor bh, byte ptr [0x1688225]
0x7ff5001c: add byte ptr [eax], al
0x7ff5001e: add cl, ch
0x7ff50020: sub byte ptr [0x2688225], bh
0x7ff50026: add byte ptr [eax], al
0x7ff50028: add cl, ch
0x7ff5002a: push ds
0x7ff5002b: cmp eax, 0x3688225
0x7ff50030: add byte ptr [eax], al
0x7ff50032: add cl, ch
0x7ff50034: adc al, 0x3d
0x7ff50036: and eax, 0x46882
0x7ff5003b: add byte ptr [eax], al
```

Ключевые моменты:

- Многократная модификация байтов по адресам в регистрах (`add byte ptr [eax], al`, `add byte ptr [esi], cl`).
- Сложная арифметика над данными (`sbb dword ptr [eax], eax`, `adc al, 0x3d`).
- Работа с байтами регистров (`cl`, `ch`, `bh`) как с ключами/масками.
- Жёстко зашитые адреса (`0x1688225`, `0x2688225`, `0x3688225`) — глобальные структуры/таблицы в памяти.
- `push ds` — манипуляция сегментом данных.
- `and eax, 0x46882` — маскирование части значения/адреса.

Интерпретация:

- Регион содержит код, который активно патчит и модифицирует память по регистровым и константным адресам.
- Поведение похоже на **декодер/патчер конфигов/структур**, а не на PowerShell‑интерпретатор.

### Регион 0x7ff60000: структурированные данные и `in al, dx`

Фрагмент:

```text
3672 powershell.exe 0x7ff60000 0x7ffaffff VadS PAGE_EXECUTE_READWRITE ...

ec ff ff ff 04 00 00 00 01 00 00 00 00 00 08 01
1c 00 00 00 15 00 0e 00 0e 00 00 00 98 26 a6 66
00 10 7f 66 dc 70 81 66 2c 30 7f 66 00 00 00 00
00 00 00 00 10 00 f5 7f 1a 00 f5 7f 24 00 f5 7f

0x7ff60000: in al, dx
```

Наблюдения:

- Основная часть — структурированные значения, похожие на таблицы/смещения/указатели.
- Единственная инструкция — `in al, dx`:
  - чтение байта из I/O‑порта (номер порта в `DX`, значение попадает в `AL`).

Интерпретация:

- Этот регион содержит данные, с которыми работает шеллкод, плюс низкоуровневую операцию ввода с порта.
- В сочетании с `out dx, al` из другого региона подтверждается использование **аппаратных портов ввода‑вывода**, что нетипично для обычного PowerShell.

## TTP / MITRE

Кейс коррелирует с:

- **T1059.001 – PowerShell** — злоупотребление PowerShell как интерпретатором команд и скриптов.  
- **T1620 – Reflective Code Loading / In-Memory Execution** — исполнение нагрузки в памяти без записи PE‑файла на диск.  
- **T1055 – Process Injection** — вставка шеллкода в легитимный процесс и его выполнение.  
- **T1027 – Obfuscated/Encrypted File or Information** — использование декодеров, масок, обфусцированных настроек.  
- Дополнительно, в зависимости от полезной нагрузки, возможны техники C2, разведки и т.п.

## Воздействие и реагирование

### Потенциальное воздействие

- Исполнение скрытой полезной нагрузки внутри `powershell.exe`.  
- Возможность удалённого доступа, загрузки модулей, выполнения команд без артефактов на диске.  
- Обход традиционных AV/EDR за счёт in‑memory исполнения.

### Рекомендации по реагированию

1. **Немедленные действия**:
   - завершить подозрительный процесс `powershell.exe`;
   - изолировать хост от сети;
   - сохранить дампы и журналы.

2. **Расширенное расследование**:
   - найти точку запуска PowerShell (LNK, планировщик, WMI, GPO, скрипты логон/логoff, EDR‑триггеры);
   - проанализировать PowerShell Operational, Security, Sysmon/EDR‑логи;
   - поиск аналогичных RWX‑регионов в других процессах.

3. **Усиление детектирования**:
   - правила на:
     - длительно работающие/скрытые `powershell.exe`;
     - наличие RWX‑регионов в скриптовых процессах;
     - использование `in/out`, `push ds/es`, `retf` в пользовательских процессах.
   - включение AMSI/ScriptBlock Logging и детекторов обфускации там, где возможно.

4. **Обучение и отработка**:
   - использовать кейс как тренировочный для SOC/DFIR;
   - встраивать его в плейбуки с PowerShell‑атаками.

## Вывод

Анализ RWX‑областей памяти процесса `powershell.exe` показал:

- наличие низкоуровневого x86‑кода, декодера и патчера данных;  
- работу с жёстко зашитыми адресами и структурированными таблицами;  
- использование нестандартных для PowerShell инструкций (`push ds/es`, `retf`, `in/out dx, al`);  
- активное модифицирование памяти по разным адресам.

Поведение не совпадает с ожидаемой работой PowerShell‑интерпретатора и однозначно указывает на присутствие **инжектированного шеллкода**, исполняемого в памяти легитимного процесса.

Этот кейс можно использовать:

- как боевой пример для собеседований и тренингов Blue Team/DFIR;  
- как шаблон оформления отчётов об инцидентах, связанных с PowerShell‑based in‑memory атаками.
